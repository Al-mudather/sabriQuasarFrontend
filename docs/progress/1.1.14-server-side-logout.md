- **Date:** 2026-06-16

## Goal
The backend added a `logoutUser` GraphQL mutation that clears the Django session
and expires the **HttpOnly `sessionid` cookie** left behind by social-login
sessions. JWTs are stateless, so the client must still discard its own tokens —
which our logout already did. This task adds the missing server-session step.

## What we already had (3 of the doc's 4 steps)
Our single logout routine — `useLogout()` → `auth.logOut()` → `logOutAction()` —
already did: `revokeToken` (refresh-token revoke), local-storage wipe, and Apollo
cache/WS teardown. **Only the server-session `logoutUser` call was missing.**

## What we built
- **`src/apollo/client.js`** — `createUploadLink` now sets
  `credentials: 'same-origin'`. REQUIRED: the `sessionid` cookie is HttpOnly, so
  only the server can delete it (via `Set-Cookie`), and only if the browser sends
  it on the request. App + `/api/graphql/` share `stc.training` in prod → same-origin.
- **`src/graphql/account_management/mutation/LogoutUser.js`** (new) —
  `mutation LogoutUser { logoutUser { success errors } }`. `npm run codegen`
  regenerated `generated.ts`; narrowed `LogoutUserResult` / `LogoutUserVariables`
  aliases added to `src/types/auth/types.ts`.
- **`src/stores/auth.ts`** — new `LOGOUT_SERVER_SESSION()` action; `logOutAction`
  now calls it **first** (before the refresh-token revoke and before storage is
  wiped), best-effort + time-boxed (`Promise.race`, 2.5s) like the existing revoke,
  and unconditionally (the mutation is a safe no-op when already logged out). Typed
  via `apolloClient.mutate<LogoutUserResult, LogoutUserVariables>(...)`.

Final order in `logOutAction`: **logoutUser → revokeToken → resetApolloSession →
revokeGoogle → purgeClientStorage → deleteData**. Post-logout redirect stays
**Home** (existing pattern; the global guard handles gating).

## Dev proxy so localhost clears cookies (`55498ca`)
Originally cookie-clearing couldn't be tested locally: the app ran on `localhost`
talking cross-origin to `stc.training`, so `sessionid`/`csrftoken` were
`stc.training`-domain cookies — a `localhost` script can't delete a cross-domain
(HttpOnly) cookie, and a SameSite=Lax cookie isn't sent on a cross-site `fetch`,
so `logoutUser` had nothing to clear. This is a browser boundary, not a bug;
prod works because the SPA is served same-site as the API.

Fix: Quasar `devServer.proxy` forwards `/api → stc.training`
(`changeOrigin` + `cookieDomainRewrite:''`), and the GraphQL endpoint uses a
RELATIVE path in dev (`src/apollo/client.js`, gated on `process.env.DEV`). The
browser now sees the API same-origin (`localhost`), so session cookies become
first-party `localhost` cookies that logout clears. Media/other URLs keep
`API_URI` absolute; the prod build still bakes the absolute URL. **Requires a
fresh login after the change** (cookies re-set on `localhost`).

## Every logout path calls the mutation
Audited all logout triggers — `AppHeader`, `UserLayout` (sidebar + drawer),
`RegisterationCode` link, the `Home/*` nav menus, and the `App.vue` session-expiry
auto-logout. **All** funnel into `auth.logOutAction()`, where `logoutUser` is
step 1, so no logout path can skip it.

# 1.1.10 — Auth: fix laggy / silent Google sign-in

- **Date:** 2026-06-16
- **Status:** 🟡 partial (cancel path live-verified; full success loop needs a real login)
- **Commit:** `e5ccaca` · **tag:** `google-signin-fix`

## Goal
Users reported that clicking "Login using Google" sometimes did nothing — no
spinner, no popup, no feedback ("did I click? where's the progress?"). When it
did work it felt laggy.

## Root cause
The app uses Google Identity Services (GIS) `initTokenClient` (modern, correct)
wrapped in a `window.hello` shim (`boot/socail_auth.js` + `boot/hello.js`). Two
problems:
1. **Broken user-gesture chain.** GIS was loaded lazily on first click, then
   `requestAccessToken()` ran inside the script-load `.then()`. Google's docs
   require `requestAccessToken()` to fire inside the synchronous click handler,
   or the browser blocks/defers the popup. So the **first** click often did
   nothing; a second click (script cached) worked — the intermittency users saw.
2. **Silent failures.** Every error path was swallowed (`.catch(() => {})`,
   one-message-only mutation handling, empty-token no-ops) → no feedback.

## What we built
- `index.html` — preconnect + preload `accounts.google.com/gsi/client` so GIS is
  ready before the click and the popup opens in-gesture.
- `boot/socail_auth.js` — build `initTokenClient` once and cache it; the click
  only calls `requestAccessToken()`. `error_callback` now distinguishes a user
  cancel from a real error.
- `GoogleAuthentication.vue` — spinner flips the instant the button is clicked
  (covers popup → token → mutation); every path now shows a bottom toast
  (cancel = info, blocked/error/empty-token = danger); 45s backstop timer.
- i18n — 3 new keys in `ar` + `en`.

## Verification
`tsc` clean, `quasar build` OK, headless mount of `/account/login` clean. The
**cancel path was live-confirmed** by the user ("Sign-in cancelled." toast). The
full choose-account → logged-in → redirect loop can't run headless — still wants
one real login.

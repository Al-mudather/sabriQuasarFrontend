# 1.1.28 — Fix "Go to lesson" bouncing enrolled users to login (dead classroom link)

**Status:** ✅ shipped (`tsc --noEmit` + `quasar build` green; root cause is a stale hard-link, fixed in all 4 call sites)

## Symptom
On the Courses page, an **enrolled** course's **"Go to lesson"** button sent the
(already-logged-in) user to the **login page** instead of into the classroom.

## Root cause
The button navigated via a hard-coded `window.location.href`:

```js
`${location.origin}/classroom/#/class/${pk}/`
```

That URL points at a **separate hash-based classroom app that no longer exists**.
The current SPA serves the classroom in-app at `/classroom/class/:coursePk`
(history mode, no `/#/` prefix, guarded by `requireAuthentication`). The dead URL
doesn't match the router, so it falls through to the auth guard / not-found and
bounces to login.

## Fix (frontend only)
Replaced the dead hard-link with a typed in-app navigation in **all four** places
that used it:

```js
router.push({ name: 'classroom-shell', params: { coursePk: String(pk) } })
```

`classroom-shell` (`class/:coursePk(\d+)`) takes the digits-only course **pk** and
auto-resolves the first lesson (per 1.1.25).

| File | Site |
|---|---|
| `components/utils/courseCard.vue` | Courses page card (reported case) — kept its spinner via `withMinDuration` |
| `components/MyCourses/courseCard.vue` | My Courses card (added `useRouter`) |
| `components/notification_management/NotificationCard.vue` | Q&A notification → classroom |
| `pages/notification_management/Notification.vue` | notifications page → classroom |

The Courses-page fix was done by a Sonnet sub-agent; the other three same-bug
sites were found + fixed here.

## Verification
- `grep` confirms **no** `/classroom/#/class/` navigations remain (only comments).
- `npx tsc --noEmit -p tsconfig.json` → clean · `npx quasar build -m spa` → succeeds.
- Live click needs a logged-in enrolled session (HMR-verified by the user).

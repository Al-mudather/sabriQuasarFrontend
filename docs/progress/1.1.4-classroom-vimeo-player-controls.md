# 1.1.4 — Classroom: Vimeo player control bar

- **Date:** 2026-06-16
- **Status:** ✅ done (Vimeo) · ⛔ VdoCipher parked
- **Commit:** `fa49062`

## Goal
Migrate the [1.1.3](1.1.3-classroom-hls-player-controls.md) custom control bar
to the other video providers.

## What we built (Vimeo)
- `src/composables/classroom/useVimeoPlayer.ts` — same shape as
  `useVideoJsPlayer`, driving `@vimeo/player` with `controls: false`; bridges
  its events (timeupdate/progress/volumechange/ratechange/buffer/ended) to the
  shared reactive state and re-emits begin/progress/complete/error.
- `src/components/classroom/player/VimeoPlayer.vue` — rewritten as a shell
  (16:9 stage + the **shared** `ClassroomVideoControls`), with fullscreen +
  keyboard, mirroring the HLS player.
- `ClassroomContentView.vue` relaxes the 16:9 box for `vimeo` too.

## The sizing bug (and fix)
First cut used Vimeo's `responsive: true`, which injects a `padding:177.78%`
aspect-ratio wrapper and absolutely-positions the iframe inside **that** — so
the iframe ballooned (~604×1073) and the stage clipped it to a thin slice.
**Fix:** drop `responsive` (no wrapper injected) and force the iframe to fill
the stage via CSS (`!important`, `padding:0`) — same technique the VdoCipher
player already uses. Verified live (video sits correctly in the stage).

## VdoCipher — parked (needs backend)
The current `VdoCipherPlayer.vue` injects a **pre-signed, cross-origin iframe
blob** (`cipher_iframe`) via `v-html`. From the frontend we can neither (a) hide
its native controls (baked into the server-side embed config) nor (b) read/drive
playback without its `vdo.js` postMessage API + a different init (OTP +
playbackInfo). Both require backend changes. Once the backend exposes that, the
frontend work is small: another `useVdoCipherPlayer` controller in the same
shape, reusing `ClassroomVideoControls`. **Roadmap item.**

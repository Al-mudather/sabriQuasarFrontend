# 1.1.13 — Forms: RTL/LTR direction polish + password visibility

- **Date:** 2026-06-16
- **Status:** ✅ done
- **Commits:** `9093fe7` (`userinfo-bidi-direction`), `4c8691e` (`login-form-direction-eye`)

## Goal
A cluster of bidirectional-text (bidi) issues on the cart user-info form and the
login form, reported during live testing, plus a missing password-reveal toggle.

## Cart user-info (`9093fe7`)
`src/components/ShoppingCard/userInformation.vue` — these were **direction**
problems, not field bugs (there was no space-blocking code anywhere):
- **Name "can't type a space":** a Latin name in an RTL field made the trailing
  space collapse visually and the caret jump. `dir="auto"` flips it LTR for
  Latin, RTL for Arabic.
- **Phone `+966` on the wrong side:** the row is now `dir="ltr"` → `+966` on the
  left, number to its right, digits read LTR.
- Telegram + email inputs set `dir="ltr"` (their content is LTR) while the Arabic
  labels stay right-aligned.

## Login form (`4c8691e`)
1. **Password eye toggle** — `DsInput` now renders a show/hide eye for
   `type="password"` (swaps `password` <-> `text`). Reusable, so signup and
   password-reset get it too. aria-labeled + i18n (ar/en).
2. **Email + password `dir="ltr"`** — entered in Latin regardless of UI language.
3. **Locale-aware auth shell** — `Account.vue` hardcoded `dir="rtl"` on `<main>`,
   forcing RTL even in English (mangled the fine-print/subtitle punctuation). Now
   `:dir` follows the locale (English ⇒ `ltr`); the Arabic branding panel stays
   `dir="rtl"`.

## Verification
`tsc` clean, `quasar build` OK, headless mounts confirm: eye toggle renders, both
login inputs carry `dir="ltr"`, the auth shell `dir` is dynamic, no runtime errors.

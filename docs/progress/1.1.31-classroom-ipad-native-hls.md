# 1.1.31 — Fix classroom video not playing on iPad/iPhone (native HLS)

**Status:** ✅ shipped (`tsc` + `quasar build` green; desktop verified; iPad needs on-device check)

## Symptom
Self-hosted (Al-Hasif) classroom videos failed to play on iPads — including in
**Chrome on iPad**, which is why "use Chrome instead of Safari" didn't help.

## Root cause (it's the player, not the backend)
The stream is **plain `AES-128` HLS** — confirmed by pulling a real manifest:

```
#EXT-X-KEY:METHOD=AES-128,URI="https://video.cdn1.stc.training/api/video/key/{uuid}/",IV=0x…
```

That is standard HLS encryption (NOT DRM, NOT ClearKey). Apple's WebKit plays it
**natively**. But the player forced video.js's JavaScript/MSE engine:

```js
vhs: { overrideNative: true }, nativeAudioTracks: false, nativeVideoTracks: false
```

On iOS/iPadOS every browser is WebKit, and forcing MSE there breaks playback —
iPad needs the **native** HLS path. The `keySystems: {'org.w3.clearkey': {}}`
config was a misnomer: an AES-128 manifest carries no ClearKey EME signaling, so
it was dead config (the `videojs-contrib-eme` plugin did nothing).

## Fix (frontend only)
Let WebKit play natively; keep VHS only where there's no native HLS:

```js
const nativeHls = videojs.browser.IS_ANY_SAFARI || videojs.browser.IS_IPAD
html5: {
  nativeAudioTracks: nativeHls,
  nativeVideoTracks: nativeHls,
  vhs: { cacheEncryptionKeys: true, overrideNative: !nativeHls },
}
```
Removed the dead ClearKey `keySystems` + the `videojs-contrib-eme` import.

Applied to **both** live players:
- `src/composables/classroom/useVideoJsPlayer.ts` (classroom `TypeHasifHlsPlayer`)
- `src/utils/AlhasifVideoPlayer.vue` (course-preview dialog) — same bug.

## Why NOT `canPlayType` (a trap we hit and backed out of)
First attempt detected native HLS via
`video.canPlayType('application/vnd.apple.mpegurl')`. A headless harness against
the real manifest caught a **desktop regression**: Chrome returns **`"maybe"`**
for that MIME (a false positive), which disabled VHS on desktop → **0 quality
levels → the quality selector disappeared**. `videojs.browser.IS_ANY_SAFARI`
returns `false` on Chrome (correct) and the harness then reported **8 quality
levels**. So the detector is video.js's browser flags, not `canPlayType`.

## Notes
- On iPad the native player shows iOS's own controls; the quality menu may
  collapse to "Auto" (native HLS exposes fewer levels) — acceptable.
- The inherent sharpness ceiling is the **source bitrate**: the top rendition is
  ~300 kbps at 1200×1920, so even max quality is soft. Sharper video needs a
  **higher-bitrate re-encode on the backend** — the player can only pick among
  the renditions that exist.
- On-device iPad verification is still pending (headless macOS can't exercise the
  Apple native-HLS path).

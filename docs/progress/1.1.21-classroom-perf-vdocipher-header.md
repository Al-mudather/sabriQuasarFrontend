# 1.1.21 — Classroom performance, VdoCipher reliability & header UX

**Status:** ✅ shipped (build + `tsc --noEmit` + `codegen:check` green; layout smoke-tested at mobile width)

Students reported the classroom felt slow, videos sometimes didn't load, and the
VdoCipher fullscreen button did nothing. This batch addresses the real causes and
records the one "issue" that turned out **not** to be a bug.

## What changed

### Performance (`classroom-perf-single-fetch`)
- **One fetch per lesson.** `useCurrentContent` ran the lesson query twice — a
  `useQuery` plus a TTL `watch` that called `refetch()` on mount. The second fetch
  re-loaded the VdoCipher iframe and burned its single-use OTP. Now a single
  `network-only` fetch (keeps the OTP fresh on revisit without a second call).
- **Parallel enrollment.** `useCourseBootstrap` keyed the enrollment query on the
  resolved course's pk, so it waited for the course query to finish. Keyed it on the
  route `coursePk` instead → course + enrollment fetch in parallel. Removed the
  post-resolve refetch watch; added `refreshEnrollment()` for the cheap ring update.
- **Slimmer queries.** Dropped fields the classroom never reads from
  `GetCourseByIDSlim` / `GetCourseUnitContents`; quiz questions go `cache-first`.

### VdoCipher reliability + fullscreen (`vdocipher-single-injection-fullscreen`)
- The `cipher_iframe` blob carries a **single-use OTP**. Switched from `v-html`
  (re-evaluates on every render) to **imperative one-time injection** into a mount
  node, guarded by an `injected` flag. The player now loads exactly once; a real
  lesson change remounts it (keyed on content pk) with a fresh OTP. Verified via a
  driven browser session: a single `embed/index.html` request, no double-load.
- **Fullscreen** now works: the backend blob's `allow` attribute omitted
  `fullscreen`, which makes Chromium ignore the legacy `allowFullScreen`. We rewrite
  the `<iframe>` to delegate `encrypted-media; fullscreen; picture-in-picture`.
- HLS player: `preload: 'metadata'` + `reset()` between source swaps so a previous
  lesson's segment loaders can't stall the next.

### Header / layout UX (`classroom-header-mobile-ux`)
- Header shows the **current lesson name, centered** (not the course name); removed
  the language switcher from the classroom header and the now-redundant content-area
  `<h2>` title.
- Player is keyed on `current.pk` (stable iframe across resize) and **centered at
  mobile width**.

## The VdoCipher 6007 finding (NOT a bug — do not re-chase)
Error `6007 Authentication failed` reproduces **only in DevTools device-emulation
mode**, and is identical code/lesson to the desktop case that plays fine. Proven by
A/B: same lesson at full window plays; toggling the iPhone device toolbar produces a
`POST license.vdocipher.com/auth 403`. Cause: device-emulation **spoofs an iOS
User-Agent**, so VdoCipher's multi-DRM auth selects FairPlay (iOS) while the machine
actually sends a **Widevine** (desktop Chrome) handshake — an impossible combination
no real device has → 403 → 6007. Real iPhones (FairPlay) and real Android (Widevine)
are unaffected. **To test small screens, drag the real window narrow — don't use the
device toolbar for DRM video.** The `injectScriptAdjust.js` AbortErrors in the console
are an unrelated browser extension (present even when playback succeeds).

### Final resolution (researched, confirmed, no code change)
A multi-source review (VdoCipher docs, Stack Overflow, GitHub, Chrome DevTools docs,
MDN, Mux/Gumlet/Bitmovin) plus a decisive **A/B/C** confirmed the mechanism beyond
doubt:
- **iPhone preset (iOS UA) → 6007/403** ❌ · **Samsung preset (Android UA) → plays** ✅
  · **real desktop → plays** ✅. Same machine, same lesson — only the UA changes.
- VdoCipher is multi-DRM and selects the scheme by User-Agent: **FairPlay for iOS,
  Widevine for Chrome/Android**. **FairPlay does not run in Chrome** (Apple-only), and
  DevTools device presets override the UA but **not** the CDM. So an iOS-UA emulation
  asks desktop Chrome for a FairPlay key it can never get → 403 → 6007. Android-UA asks
  for Widevine, which desktop Chrome has → works.
- **Decision: no fix and no backend change.** The frontend is correct; this is a
  DevTools-emulation artifact, not a defect. There is nothing to change.
- **Open verification (product, not code):** desktop Chrome can never exercise the real
  FairPlay path, so the only way to confirm real iPhone students are covered is to open
  a lesson on an **actual iPhone (Safari)**. If real iPhones play, FairPlay is
  provisioned on the VdoCipher account and everything is fine; if they don't, FairPlay
  must be enabled on the account (a VdoCipher-dashboard setting, outside this codebase).

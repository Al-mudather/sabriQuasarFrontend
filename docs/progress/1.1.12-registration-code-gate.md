# 1.1.12 — Auth: enforce registration-code gate globally; fix login bypass

- **Date:** 2026-06-16
- **Status:** 🟡 partial (guest + code-traced paths verified; authenticated paths need a live login)
- **Commit:** `bc3725e` · **tag:** `registration-code-gate`

## Goal
Users reported that after Google sign-in they were never shown the registration
code. Business rule: an authenticated user **not** linked to a PyramidAffiliate
must be forced to enter a registration code before using the platform / completing
a purchase. Guests (no token) must stay completely unrestricted on public pages.

## Investigation (two parallel read-only audits)
Ran two `sonnet` sub-agents — one on the login-side gate, one on the payment-side
gate.

**Root cause (login):** a regression from [1.1.8](1.1.8-course-redirect-and-clean-urls.md)
(`ee21b42`). `checkRegistrationCode()` returned early on a `?redirect=` param
**before** the PyramidAffiliate check ran. Since `requireAuthentication` appends
`?redirect=` to nearly every protected nav, the gate effectively never fired on
Google login (the primary path). Enforcement also lived **only** as a bypassable
post-login `router.push` — there was **no global guard**.

**Payment side:** three checkpoints existed but were brittle — `cache-first`
Apollo policy, inconsistent error string-matching (`===` vs `.includes()`), one
not `await`ed. Best-effort UX, not hard enforcement (backend enforces server-side).

## What we built (defense-in-depth)
- **`router/index.js`** — global `beforeEach` gate. The first line short-circuits
  on **no token** (guests pass straight through — public pages stay open).
  Authenticated users without a PyramidAffiliate are routed to `registeration-code`
  with `?redirect=` preserving their destination. Exempt list (auth/legal/error/
  gate pages) prevents trapping them. This single guard closes **every** login
  leak at once.
- **`stores/pyramid.ts`** — cached `hasPlatformAccess` + `verifyPlatformAccess()`
  (**network-only**, robust matching, **fails open** on transient/network errors
  so a blip never locks anyone out), `markPlatformAccessGranted()`,
  `resetPlatformAccess()`. Auto-resets on logout (`$reset`) and login.
- **`RegisterationCode.vue`** — on success, mark access granted + route to the
  stashed `?redirect=` (land on the course they tried to enrol in), not always Home.
- **`Login.vue` / `GoogleAuthentication.vue`** — reset the flag on login
  (account-switch safety).
- **`ShoppingCart.vue` / `payment.vue`** — payment-side backstop now calls the
  store's `verifyPlatformAccess(true)` (network-only) instead of brittle inline
  checks; dropped dead imports.

## New behavior
Gate fires only when **both**: a token is present **AND** the user has no code.
Guests (no token) and users-with-a-code are unaffected.

## Verification
`tsc` clean, `quasar build` OK. Headless (no token = guest) loads of `/`, `/courses`,
`/account/login`, `/account/registerationCode` all render their own content with
no redirect and no runtime errors — **guests confirmed unrestricted**. The
authenticated paths (with-code browses freely; no-code forced to the gate, then
lands on the course after entering it) can't run headless — **needs a live login**.

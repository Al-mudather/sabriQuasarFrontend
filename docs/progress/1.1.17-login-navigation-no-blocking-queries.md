# 1.1.17 — Auth: remove blocking queries from post-login navigation (real stuck-login fix)

- **Date:** 2026-06-16
- **Status:** ✅ done (build/headless-verified; live login confirmation welcome)
- **Tag:** `login-nav-no-blocking-queries`

## Symptom (recurred after 1.1.16)
After a successful login (Google or email) the big spinner appeared, then
**stopped with no navigation** — the user had to click login a **second** time,
which worked. 1.1.16 removed the un-awaited (`void`) pushes but the underlying
latency + a verdict bug survived.

## Root cause — two defects
1. **Navigation blocked on sequential network round-trips.**
   `navigateAfterLogin()` ran `isUserEnrolled()` (up to 3s) **before** pushing,
   and the global router guard then ran `verifyPlatformAccess()` (up to 5s) on
   the resulting navigation — up to **~8s of stacked queries** on a *cold* first
   attempt against the prod backend. The second click was fast because the
   queries were warm and `hasPlatformAccess` was cached. That is exactly the
   "spinner stops, nothing happens, click again" signature.
2. **`verifyPlatformAccess` misread a transient error as "no code."**
   The Apollo client default is `errorPolicy: 'all'`, so a GraphQL error on the
   **first authenticated request right after login** does NOT throw — it returns
   `data: null` with `res.errors` populated. The old line
   `res.data?.checkPyramidAffiliate != null` then evaluated to `false`, so the
   guard could bounce a legitimate user to `registeration-code`. The `catch`
   fail-open only caught *network* throws, never this case.

## Fix
- **`navigateAfterLogin`** (`GoogleAuthentication.vue`, `Login.vue`,
  `FacebookAuthentication.vue`): navigate **immediately** to `?redirect=` or
  `Home` — no blocking pre-queries. Uses `router.replace` so the login page
  leaves history. Registration-code gating stays with the global guard (single
  source of truth). Removed the now-unused `isUserEnrolled` helper +
  `AllEnrollmentsForCurrentUser` import from all three.
- **`verifyPlatformAccess`** (`src/stores/pyramid.ts`): explicit
  `errorPolicy: 'all'`, and **fail OPEN** when `res.errors` is present (do NOT
  cache a verdict — re-check next navigation). Only a clean response decides:
  `data.checkPyramidAffiliate` null → no code, non-null → has access. The
  network/throw `catch` also fails open without caching.

Net: the happy path is now a **single** guard query (usually sub-second), and a
transient first-call error can no longer divert a valid user.

## Tradeoff
Dropped the "enrolled users land on `/myCourses`" routing — everyone now lands
on `Home` (or their original `redirect`). It was already unreliable (the 3s
timeout sent enrolled users to Home on a slow network) and was a direct
contributor to the hang. If wanted back, its proper home is the `Home` page,
not a blocking gate on the login path.

## Verification
`tsc --noEmit` clean, `quasar build -m spa` succeeds, headless login mount
renders with no new runtime errors (only the expected OneSignal
localhost-origin warning). Real Google/email success redirect needs a live
login — takes effect on stc.training after a deploy.

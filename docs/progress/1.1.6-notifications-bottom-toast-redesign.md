# 1.1.6 — Notifications: bottom toast redesign

- **Date:** 2026-06-16
- **Status:** ✅ done (live-verified via login)
- **Commit:** `16a5323`

## Goal
The notification system was fragmented — only 3 files used the DS `toast`
helper, while **37 files** called raw `$q.notify(...)`, and all defaulted to
**top**. Unify it, move to the **bottom**, and restyle to match the new cream
design language.

## Direction chosen
**"Ivory + accent"** (over "solid brand"): cream card, ink text, a tinted round
icon chip, hairline border, soft shadow, and a per-type colored progress line.

## What we built
- `src/css/ds-toast.scss` (imported in `app.scss`) — global toast styling keyed
  off `ds-toast` + `ds-toast--<type>` classes. Global because Quasar renders
  notifications outside component scope.
- `src/boot/notify.js` — `Notify.setDefaults({ position: 'bottom' })` plus
  `Notify.registerType` mapping Quasar's `positive/negative/warning/info` to the
  `ds-toast` classes, so **raw `$q.notify` calls adopt the look without editing
  each site**. (Added to the `boot` array in `quasar.config.js`.)
- `src/design-system/toast.js` — defers color to the shared CSS classes,
  defaults to bottom.
- **Codemod:** `position: 'top'` → `'bottom'` across all 35 notify-using files.

## A bug we caught
The codemod initially also flipped `DsTooltip.vue`'s default `position: 'top'` —
but that's a **tooltip placement prop**, not a notification. Reverted, then
re-scanned every `position: 'bottom'` to confirm all are genuinely `$q.notify`
calls. Lesson: a blind string codemod needs a per-hit context check.

## Verification
`quasar build` succeeds; `.ds-toast--*` CSS + the `registerType` boot confirmed
present in the built bundle. **Live-verified** end-to-end via the login toast in
[1.1.7](1.1.7-login-toast-feedback.md).

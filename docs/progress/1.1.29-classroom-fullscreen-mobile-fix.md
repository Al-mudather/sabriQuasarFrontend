# 1.1.29 — Fix dead fullscreen button on mobile (iOS element-fullscreen)

**Status:** ✅ shipped (`tsc --noEmit` + `quasar build` green)

## Symptom
In the classroom video player, the **fullscreen button did nothing on phones /
small screens** — tap, no response. On desktop it worked. Reported as a
regression ("it worked before").

## Root cause
Both custom players requested fullscreen on their `<div>` shell:

```js
void el.requestFullscreen?.()   // el is a <div>
```

**iOS Safari does not implement the Fullscreen API on non-`<video>` elements**, so
`div.requestFullscreen` is `undefined` and the optional-chain silently no-ops.
Desktop Chrome implements it on any element → worked there. Hence
"works on desktop, dead on phone."

Two latent gaps alongside it:
- No **webkit-prefixed** fallback (`webkitRequestFullscreen` / `webkitExitFullscreen` /
  `webkitFullscreenElement`) for older Android WebKit / Safari.
- Only `fullscreenchange` was listened to, not `webkitfullscreenchange`, so the
  `isFullscreen` UI state could desync on WebKit even when fullscreen succeeded.

The 1.1.27 auto-hide work was **not** the cause — `data-controls-hidden` is only
set in fullscreen (`(isFullscreen && !controlsVisible) || null`), so the
windowed-mode tap path was never blocked. The "worked before" predates the
1.1.21 shell-fullscreen change.

## Fix (frontend only)
New typed helper `src/composables/classroom/useFullscreen.ts` — vendor-prefixed
APIs go through narrow local interfaces (`FsElement`, `FsVideoElement`,
`FsDocument`); **no `as any`, no `@ts-expect-error`** (per CLAUDE.md). Request
waterfall: W3C → `webkitRequestFullscreen` → iOS `video.webkitEnterFullscreen()`.
`exitFullscreen()` and `getFullscreenElement()` mirror the prefix waterfall;
`FS_CHANGE_EVENTS` registers both change events.

| File | Change |
|---|---|
| `composables/classroom/useFullscreen.ts` | **new** cross-browser helper |
| `player/TypeHasifHlsPlayer.vue` | `requestFullscreen(el, videoEl.value)` — passes the real `<video>` as the iOS fallback; `getFullscreenElement()` + both change-events |
| `player/VimeoPlayer.vue` | same fullscreen fix (no video fallback — iframe → Vimeo's own iOS fullscreen); also ported in the missing 1.1.27 auto-hide controls for parity |

## Caveat
On iOS the HLS player goes fullscreen via the **native iOS video player** (its own
controls, not our custom bar) — the only fullscreen path iOS allows for
self-hosted video. Acceptable, and the button now works.

## Verification
- `npx tsc --noEmit -p tsconfig.json` → clean · `npx quasar build -m spa` → succeeds.
- Note: `tsconfig` does **not** cover `src/composables/**`, so the new `.ts`
  helper was type-reviewed by hand (narrow vendor interfaces, no escape hatches).
- On-device iOS confirmation is the user's (the API gap is iOS-specific and can't
  be reproduced in desktop headless).
